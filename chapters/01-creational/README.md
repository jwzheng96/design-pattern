# 第一部分：创建型模式 (Creational Patterns)

[← 返回目录](../../README.md)

---

## 概述

创建型模式关注**对象的创建机制**，试图以适合情况的方式创建对象。创建型模式的核心思想是**将对象的创建与使用分离**，使代码更加灵活、可扩展。

### 为什么需要创建型模式？

在没有创建型模式的情况下，我们通常使用 `new` 关键字直接创建对象：

```cpp
// 简单但不灵活的对象创建
Database* db = new MySQLDatabase();
Logger* logger = new FileLogger();
```

这种方式的问题：
1. **紧耦合**：代码直接依赖具体类
2. **难以测试**：无法轻松替换为 Mock 对象
3. **不易扩展**：添加新类型需要修改多处代码
4. **配置困难**：无法在运行时决定创建哪种对象

创建型模式通过各种技术解决这些问题。

---

## 本部分包含的模式

| 模式 | 核心思想 | 适用场景 |
|-----|---------|---------|
| [单例模式](singleton/README.md) | 确保只有一个实例 | 全局配置、连接池、日志系统 |
| [工厂方法模式](factory-method/README.md) | 让子类决定创建什么 | 框架设计、插件系统 |
| [抽象工厂模式](abstract-factory/README.md) | 创建相关对象家族 | UI组件库、跨平台开发 |
| [建造者模式](builder/README.md) | 分步构建复杂对象 | 复杂配置对象、SQL构建器 |
| [原型模式](prototype/README.md) | 通过克隆创建对象 | 对象缓存、避免昂贵的初始化 |

## 模式对比

```
┌────────────────────────────────────────────────────────────────────┐
│                       创建型模式对比                                │
├──────────────┬─────────────────────────────────────────────────────┤
│    模式      │                    特点                             │
├──────────────┼─────────────────────────────────────────────────────┤
│    单例      │  控制实例数量，全局访问点                             │
├──────────────┼─────────────────────────────────────────────────────┤
│   工厂方法   │  延迟到子类决定创建什么对象                           │
├──────────────┼─────────────────────────────────────────────────────┤
│   抽象工厂   │  创建一系列相关的对象，而不指定具体类                   │
├──────────────┼─────────────────────────────────────────────────────┤
│    建造者    │  分步骤创建复杂对象，相同的构建过程可以创建不同的表示    │
├──────────────┼─────────────────────────────────────────────────────┤
│     原型     │  通过复制现有对象来创建新对象                          │
└──────────────┴─────────────────────────────────────────────────────┘
```

## 如何选择？

```
需要创建对象？
│
├─ 需要确保全局只有一个实例吗？
│   └─ 是 → 【单例模式】
│
├─ 需要创建一系列相关的对象吗？
│   └─ 是 → 【抽象工厂模式】
│
├─ 创建对象的过程很复杂，需要分步骤进行吗？
│   └─ 是 → 【建造者模式】
│
├─ 需要基于现有对象创建新对象吗？
│   └─ 是 → 【原型模式】
│
└─ 需要将创建逻辑与使用逻辑分离吗？
    └─ 是 → 【工厂方法模式】
```
